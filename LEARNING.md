## 0. 背景與約束

- 市府多局處、局處下多單位,文件風格各異,**單一規則組不可能涵蓋全府**。
- 契約內容不可外流:雲端 LLM(Claude/GPT/Gemini)不得接觸契約原文。
  語意工作只能用地端 vLLM(wbs.toml `[llm]`)。
- **格式指紋不是機密**:版面結構、編號樣式、頁尾模式、座標統計等
  去內容化資訊可以外流,可用於與雲端 AI 討論規則設計。
- 聰明的載體是地端資產:案例庫 + profile + 評測,模型只是可替換引擎。
- 開發模式沿用一期:零互動、mock 優先、每步收尾跑驗收、逐步 commit。

## 1. 系統定位與資料流

```
┌────────── 一期管線(產線)──────────┐
│ parse → … → export                  │
│   │ 低信心結果                      │
│   ▼                                 │
│ NEEDS_REVIEW 清單 ──────────────┐   │
└─────────────────────────────────┼───┘
                                  ▼
┌────────── 二期學習系統 ─────────────────────────┐
│ ① 修正介面:人標註正確答案                      │
│ ② 案例庫:輸入指紋 + 正確答案(versioned)      │
│ ③ 回流:修正 → profile 規則提議 + 回歸案例      │
│ ④ Profile 庫:市府共通 → 局處 → 單位(繼承)    │
│ ⑤ 評測器:任何變動 → 全案例重跑 → 記分卡        │
└──────────────┬──────────────────────────────────┘
               │ profile 更新
               ▼
        一期管線讀取 profile 執行(取代寫死的規則)
```

## 2. 五個元件的規格

### 2.1 案例庫(cases/)
- 一筆案例 = 一份契約的「輸入指紋 + 分層正確答案」。
- 輸入指紋:PDF sha256、頁數、各頁 layout 統計(字數/圖數/字型集合/
  頁首頁尾文字的雜湊)。**不存契約原文**;案例庫本身即可進 git。
- 正確答案分層,允許部分標註(只標到子文件層也是有效案例):
  - L1 子文件切分:每份子文件的 (title, doc_type, page range)
  - L2 章節:每章 (title, 起始 page+block)
  - L3 排除/優先級:各 section 的 priority 期望
  - L4 表格:具名表格數、跨頁表範圍
- 來源標記:`golden`(人工完整驗證)/ `corrected`(NEEDS_REVIEW 修正)
  / `auto`(管線自產、未經人工,只做煙霧用)。
- 現有 tests/golden/expected.json 遷移為第一筆 golden 案例。

### 2.2 Profile 庫(profiles/)
- 三層繼承:`base`(市府共通)→ `<局處>` → `<局處>/<單位>`。
  子層只寫與父層的差異。
- 一個 profile 描述(宣告式,非程式碼):
  - 子文件切分線索:標題錨點詞、頁尾家族 regex、封面特徵
  - 章節編號體系:壹貳參 / 一二三 / 1.1.1 / 第N條 … 的層級組合
  - 頁首頁尾帶(y 比例)與重複判定參數
  - 排除規則:哪些 doc_type / 標題模式預設 EXCLUDED
  - LLM few-shot:該單位的示範例(從案例庫引用,不重複存)
- 每次修改須附:動機案例 id + 修改前後評測分數(見 2.5)。
- 一期各 stage 改為讀 profile 執行;現行寫死規則搬進
  `profiles/base/`,行為不變是遷移的驗收條件。

### 2.3 修正介面(初期 CLI,不做 UI)
- `wbs review list`:列出專案的 NEEDS_REVIEW 與各 stage 低信心項。
- `wbs review annotate <project>`:產生一份預填的標註檔
  (管線目前的猜測 + 空白欄位),人直接編輯該檔。
- `wbs review accept <project>`:驗證標註檔 → 寫入案例庫
  → 觸發回流提議。
- 驗收:整個流程零互動指令化,可在任何裝了本工具的機器上執行。

### 2.4 回流機制(半自動)
- 輸入一筆 corrected 案例,輸出「profile 修改提議」:
  例:案例顯示頁尾「第X頁,共Y頁○○文件」未被識別 →
  提議在該單位 profile 的頁尾家族加一條 regex。
- 提議以 diff 形式落地,**人確認才合併**(初期就是 git diff + commit)。
- 每筆提議必附:套用後全案例評測分數變化。分數退步的提議直接拒絕。

### 2.5 評測器(evals/)
- `wbs eval run`:對案例庫全部案例執行管線(mock LLM),
  按層計分:子文件切分 F1、章節定位命中率、排除正確率、表格合併正確率。
- 輸出記分卡(JSON + 終端表格),按 單位/局處/全府 聚合。
- 用途:(a) 改規則/profile 的回歸剎車;(b) 換地端模型的驗收考卷;
  (c) 提示詞調整的 A/B 依據。
- 驗收:任何 PR/commit 前跑 `wbs eval run`,分數不得低於 main 基線。

## 3. 里程碑(每步完成即 commit,訊息「L<n>: <內容>」)

- **L1 案例庫格式 + 遷移**:定案例 schema(Pydantic),把 expected.json
  轉成第一筆 golden 案例;`wbs eval run` 能對這一筆跑出滿分記分卡。
- **L2 Profile 抽離**:現行寫死規則搬進 profiles/base,一期管線改讀
  profile;驗收 = 黃金測試 25 項全綠(行為不變)。
- **L3 修正流程 CLI**:review list/annotate/accept 三指令;
  用黃金契約模擬一次「故意改壞 → 標註 → accept → 案例入庫」全流程。
- **L4 回流提議**:至少支援兩類提議(頁尾家族新增、切分錨點詞新增),
  含評測分數對比。
- **L5 指紋去內容化輸出**:`wbs fingerprint <project>` 產出可外流的
  結構報告(零原文);驗收 = 人工檢視輸出不含任何契約文字。
- L6+(等真實測試案例回來再定):新單位 profile 建立流程、
  few-shot 自動抽選、模型換版驗收流程。
