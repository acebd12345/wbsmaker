# 政府契約 WBS 系統之「學習子系統」架構(徵求設計意見)

## 一、背景:現有系統

我們已建成一套地端 CLI 工具(Python),把政府採購契約 PDF 自動轉成可追溯的 WBS(工作分解結構):

- **15 階段管線**:PDF 解析 → 頁面品質偵測(亂碼頁/掃描頁隔離)→ 版面分析(頁首頁尾識別)→ 子文件切分(一份 PDF 常含契約本文/附件/需求規範書/投標須知等多份文件)→ 章節定位(block 級,支援同頁多章)→ 跨頁表格合併 → 內容分類與優先級過濾(投標須知等不該進 WBS 的整份排除)→ 地端 LLM 語意抽取工作項 → 生成三層 WBS → 匯出(互動 HTML/xlsx/csv/json)
- **已驗證**:一份 309 頁真實契約(164 頁正常文字、10 頁掃描、135 頁字型損壞亂碼、10 份子文件)跑出 508 個工作項、45 節點三層樹,付款/驗收條款可回溯到 PDF 頁碼
- **黃金回歸測試**:以該契約的人工驗證結果為 ground truth,25 項測試
- **設計哲學**:先過濾再送 LLM——規則程式處理文件工程,LLM 只碰過濾後的乾淨素材做語意抽取

## 二、限制條件(硬約束)

1. **契約內容不可外流**:雲端 LLM(GPT/Claude/Gemini)不得接觸契約原文。語意工作只能用內網 vLLM 上的開源模型(30B 級,128k context,尖峰時段 50–75 秒/次呼叫)
2. **格式指紋不視為機密**:版面結構、編號樣式、頁尾模式、座標統計等去內容化資訊可用於外部討論(本文件即屬此類)
3. **零互動**:管線任何環節不得出現互動式確認;歧義寫入 NEEDS_REVIEW 繼續執行
4. **規模**:此市政府有多個局處、局處下多個單位,公文與契約格式各異;年案量會成長,309 頁只算小案

## 三、要解的問題

現有切分/章節/排除規則是從單一機關的契約歸納的,寫死在程式裡。換一個局處的契約(例如附件標題從「附件1:」變成「附錄一、」)就會切錯。**不可能預先寫出涵蓋全府的規則**,所以系統必須「越用越聰明」:每處理一份新格式的契約、每次人工修正,都要讓下一份同類契約做得更準。

## 四、核心主張:聰明住在資料,不住在模型

由於只能用地端中型模型,學習不能依賴模型本身變強。設計主張:

- 聰明的載體是三種**地端資料資產**:案例庫、profile(格式知識)、評測記分卡
- 模型只是可替換引擎(vLLM 換權重即升級,案例庫就是換版驗收考卷)
- md 文件只在建造期指揮開發;運行期的知識全部是機器可讀資料,由確定性程式消費

## 五、架構:五個元件

```
┌────── 解析產線(已建成)──────┐
│ parse → … → export            │
│   低信心結果 → NEEDS_REVIEW ──┼──┐
└───────────────────────────────┘  │
                                   ▼
┌────── 學習子系統(本文主題)──────────────────┐
│ ① 修正介面(CLI):人編輯預填標註檔→確認入庫  │
│ ② 案例庫:輸入指紋+分層正確答案(versioned)  │
│ ③ 回流:修正 → profile 修改提議(人審 diff)  │
│ ④ Profile 庫:市府共通→局處→單位 三層繼承    │
│ ⑤ 評測器:任何變動→全案例重跑→記分卡         │
└──────────────┬────────────────────────────────┘
               │ profile 更新
               ▼
     產線各 stage 改讀 profile 執行(取代寫死規則)
```

**① 案例庫**:一筆案例 = 一份契約的「輸入指紋 + 分層正確答案」。指紋只含結構統計(頁數、各頁字數/圖數/字型集合、頁首頁尾文字雜湊),不含原文,可進 git。正確答案分層可部分標註:L1 子文件切分 / L2 章節錨點 / L3 排除與優先級 / L4 表格範圍。來源分 golden(人工完整驗證)/ corrected(修正產生)/ auto(未經人工)。

**② Profile 庫**:宣告式設定檔(非程式碼),三層繼承、子層只寫差異。內容:子文件切分錨點詞、頁尾家族 regex、章節編號體系(壹貳參/一二三/1.1.1/第N條的層級組合)、頁首頁尾帶參數、預設排除規則、該單位的 few-shot 範例引用。每次修改附動機案例 id 與前後評測分數。

**③ 修正介面**:三個指令——列出 NEEDS_REVIEW、產生預填標註檔(系統目前的猜測+空欄)、驗證並入庫。人改「答案」不改程式;一次修正的成本必須低到承辦人員可自行完成。

**④ 回流機制**:比對「系統的錯誤 vs 人給的正確答案」,自動產生 profile 修改提議(如:新增一條頁尾 regex、新增切分錨點詞),以 diff 形式呈現,**人確認才合併**,每筆提議必附套用後全案例分數變化,分數退步直接拒絕。

**⑤ 評測器**:對全案例庫執行管線(mock LLM),按層計分:子文件切分 F1、章節定位命中率、排除正確率、表格合併正確率;按單位/局處/全府聚合。三種用途:改規則的回歸剎車、換模型的驗收考卷、提示詞 A/B 的依據。

## 六、運行迴路(具體例)

1. 新局處契約進來 → 指紋比對不認得 → 保守模式解析 → 附錄切不出來 → NEEDS_REVIEW
2. 承辦編輯預填標註檔給出正確切分 → 確認指令
3. 一筆修正變三種資產:案例入庫、profile 提議(新增「附錄[一二三]、」錨點)、回歸考題
4. 提議套用前全案例重跑,舊單位不退分才合併
5. 下一份同單位契約:指紋認出 → 套該單位 profile → 直接切對;其案例同時成為 LLM few-shot 範例

## 七、明確不做(本期)

Web UI(先 CLI)、全自動回流(必經人審)、地端模型微調(案例 < 數百筆不划算)、預先設計全府 profile 分類法(等 5–10 份不同單位真實失敗案例再定形)。

## 八、想徵求意見的開放問題

1. **信心值校準**:各 stage 目前無統一信心分數。切分/章節這類結構任務,信心值怎麼定義才不會「全部都低信心」或「錯得很有信心」?
2. **Profile 表達力邊界**:宣告式 profile(regex+參數)能覆蓋多少格式變異?什麼時候該允許 profile 掛 plugin 程式碼,代價是什麼?
3. **指紋防重識別**:頁首頁尾文字以雜湊存放,但結構統計組合是否仍可能反推出機關/案件?去內容化的充分標準?
4. **案例庫治理**:正確答案本身可能錯(人也會標錯)。案例的信任等級、覆核與淘汰機制怎麼設計?
5. **多 profile 匹配衝突**:一份契約同時像兩個單位的格式時,怎麼裁決?混合格式(一份 PDF 內不同子文件來自不同單位範本)怎麼處理?
6. **微調時機**:few-shot 到什麼程度會遇到天花板?什麼訊號出現時才值得投資地端 LoRA 微調?
7. **標註 UX**:標註檔對非工程背景承辦是否可行?xlsx 標註檔 vs YAML vs 輕量 Web 表單的取捨?

# 附錄:第一輪評審意見與我方判斷(供後續討論者參考)

以下記錄本文件發出後,第一位外部評審(讀過實際程式碼)的主要意見,
以及系統原設計方(Claude)對這些意見的判斷。後續討論者可在此基礎上深入或反駁。

## 九、評審意見摘要

評審實際讀了程式碼,主要發現與主張:

1. **程式碼現況確認**(經核實全部屬實):切分規則高度耦合單一機關格式
   (s04 硬編碼「附件N/立契約人/頁尾關鍵詞/評選日期格式/框線字元」;
   s06 編號體系寫成 if/else;s09 關鍵詞表寫死);s13 呼叫了 merge LLM
   但幾乎不消費回傳;508 工作項僅約 199 進樹(孤兒項);信心值只有
   表格合併與分類兩處,切分/章節是「全有或全無」,無中間態。
2. **核心優先序修正**:「先規則外部化,再從失敗學習」。
   抽 profile 不需要等多份契約——把今日硬編碼搬進單一
   `profiles/default.toml`(內容=現行規則,行為不變,以黃金測試為驗收)
   可立即做;「等 5–10 份失敗案例」只適用於三層繼承分類法與
   自動回流提議器的定形。否則第二份契約來時仍在改 Python 而非改設定。
3. **開放問題直答**:
   - 信心值:不用模型自報,用可觀測結構訊號(錨點命中品質、頁尾家族
     連續覆蓋、三種切分依據是否同界、profile 指紋距離、覆蓋完整性)
     之乘積/最小值;輸出 stage_confidence + reasons[];用案例庫事後畫
     reliability diagram 校準;初期寧可 over-flag
   - Plugin:預設拒絕,當逃逸艙口。允許條件=同類錯誤在 ≥3 單位出現
     且宣告式無法表達;plugin 須純函式+案例測試+版本釘死+評測不退分
   - 指紋:「雜湊 ≠ 匿名」——頁尾 pattern 本身就是機關範本指紋。
     改採 need-to-know 分級存放(純結構統計可進 git;可還原之頁首頁尾
     原文/案名/機關名不進公開庫),不假裝去識別充分
   - 案例治理:四態 draft → active → disputed → retired;
     影響 profile 合併的案例須覆核;評測只跑 active
   - 混合格式當一等公民:先粗切子文件,再按子文件各自 match profile;
     文件級 top-1 與 top-2 指紋分數過近時退保守 profile + NEEDS_REVIEW
   - 微調:結構問題靠 profile 解掉後、語意抽取仍系統性漏類、
     few-shot 無增益、且有 ≥200–300 筆高品質配對時才考慮 LoRA
   - 標註:承辦不寫 YAML;xlsx 預填(系統猜測+下拉修正)→ CLI 轉機器
     JSON;「標註只改答案不改程式」死守
4. **成熟度評分**:文件工程 8/10(真金但規則內嵌)、語意抽取 6/10
   (可用但孤兒項/merge 半吊子)、可遷移性 1/10(設計有 code 無)、
   工程骨架 8/10。

## 十、原設計方對評審的判斷

**認同並採納:**

- 「先規則外部化」的優先序修正成立。原路線圖把「抽 profile」和
  「定繼承分類法」混為一談;實際上前者不需等案例,後者才需要。
  此修正等同於將學習系統里程碑 L1–L3(案例格式遷移、profile 抽離、
  修正 CLI)解除停車,僅 L4(回流提議)與繼承樹續等真實失敗案例。
- 信心值用結構訊號、plugin 當逃逸艙口、指紋分級存放、
  混合格式 subdoc 級匹配、微調時機、xlsx 標註——全部採納。

**加註兩點:**

1. **案例治理強度分期**:四態資料模型採納,但現階段常駐人力少,
   「雙人覆核」流程先簡化為「影響 profile 合併的案例需第二眼」,
   常駐標註者 ≥3 人再正式化。治理設計要跟人力現實匹配,
   否則迴路轉不動。
2. **評測的運維邊界**(評審未點破):案例庫進 git 的是「指紋+答案」,
   不含契約 PDF。因此評測只能在 PDF 實際存在的機器上跑——
   評測器設計必須把「案例引用的 PDF 不在本機」當正常情況處理
   (跳過並標示,而非報錯),且全庫跑分不是 CI 隨處可跑的東西。

**修正後的近程順序**(取代原「整體停車等案例」):

1. Profile v0:s04/s06/s09 硬編碼搬進 profiles/default.toml,
   行為不變,黃金測試全綠為驗收
2. 案例 v0:現有 golden ground truth 轉為第一筆案例(指紋+分層答案)
3. 評測 v0:對案例庫跑分層記分卡(先 1 筆案例)
4. 主幹債併行:孤兒項歸位、s13 真正消費 merge 結果、
   s11 關聯層(交付物→適用標準,亦為驗收輔助系統前置)
5. 等外單位真實失敗案例 → 三層繼承分類法 + 回流提議器定形

**留給後續討論者的問題:**

- 信心值的結構訊號組合,用乘積、最小值還是加權?過度保守
  (全部 NEEDS_REVIEW)與過度自信的實際平衡點怎麼找?
- 案例庫「答案的版本演化」:profile 改進後,舊案例的部分答案
  可能因規則語意變化而過時,如何偵測與遷移?
- 評測分數的單位聚合(單位/局處/全府)在案例數極不均衡時
  (某單位 20 筆、某單位 1 筆)如何避免誤導?

## 十一、第二輪評審摘要

同一評審對第十節的回應,要點:

1. **抓到原設計方的失誤**:第十節宣稱「L1–L3 解停車」,但列近程順序時
   漏掉了 L3 修正介面。無 annotate/accept 則案例庫永遠只有 golden 一筆,
   學習迴路名存實亡。近程須補 Correct CLI v0(可以很醜,只支援 L1 切分表)。
2. **Profile v0 驗收補強**:goldtest 綠不夠,另需 (a) stages 禁止再出現
   業務 regex 字面量(allowlist 僅限通用工具);(b) default.toml 欄位與
   現行程式的一一對照表;(c) 突變測試——故意改壞一條錨點,goldtest 必須紅,
   證明程式真的在讀 profile。
3. **案例 v0 與評測 v0 應同週交付**:沒有評測器消費的案例只是換目錄放的
   expected.json。
4. **主幹債移出關鍵路徑**:孤兒項/s13/s11 與學習閉環解耦、並行;
   s11 是支線二前置但學習閉環不依賴它。
5. **三個新問題的直答**:
   - 信心:決策用 min 閘門,排序/除錯用加權 breakdown;目標假陰性 <5%、
     初期假陽性容忍 20–30%;先固定閾值 T=0.6,5+ 案例後用
     reliability diagram 校準閾值,不調公式形狀
   - 答案版本演化:案例釘 answer_schema_version;profile 合併後某案例
     分數無故大變 → 標 stale_suspect → 人工裁定更新或 retired,
     禁止靜默改寫 golden;記分卡雙欄——「vs 凍結標註」(回歸剎車)與
     「vs 最新裁定」(真實能力)分開,避免改對了卻被舊標註罰分
   - 不均衡聚合:macro 與 micro 並陳;單位案例 <3 標 low_n 不進排名;
     合併剎車雙保險——任何 active 案例不得跌破門檻(保護稀有單位)
     且 macro 不退步(保護整體);「全府 +0.5% 但唯一交通局案例 -30%
     → 拒絕合併」
6. **運維細節落地**:案例記 pdf_sha256 + 可選 pdf_locator(路徑不進 git);
   本機有 PDF 先核 SHA,不符 FAIL(防同名不同檔);缺 PDF → SKIP 並計入
   記分卡 skipped 欄;CI 只跑合成小 PDF + schema 驗證,全庫記分卡為
   夜間/工作站任務。
7. **治理落地**:promote 到 active 需明確指令/角色,日常 accept 只進
   draft;合併 profile 必須引用 active 案例 id;人力 <3 時允許同人
   隔日 self-review,但 log 記 reviewer=same 可過濾。
8. **文件小洞**:profile 欄位級 schema 未定;s04 頁尾家族 pattern
   白名單必須進 profile(s03 band 參數可暫留程式);s02 預留
   garbled_signals 欄位(雙語附錄誤傷);「保守模式」未定義——最少集為
   「只用 default profile + 更低信心閾值 + 禁止長段 forward-fill 切分
   + 產完整預填標註檔」;回流提議 v0–v3 就是人看 diff 手改 profile,
   評測只負責剎車,勿高估自動生成;學習與主幹債分兩套 DoD。
9. **收斂宣告**:設計共識已足夠開 Profile v0;討論應下沉到資料形狀
   (schema/欄位/skip 語意),不要再辯論五元件是否該存在。

## 十二、原設計方對第二輪的判斷(最終共識)

- **失誤承認**:近程順序漏列修正 CLI 屬實,已採納補回。
  修正後近程:① Profile v0 → ② 案例 schema + 11108 案例 v0 與
  ③ 評測 v0 同週 → ④ Correct CLI v0(list/annotate/accept→draft)
  → 主幹債並行不阻擋 → 外單位失敗後才做繼承+回流。
- **全數採納**:突變測試、雙欄記分卡、合併剎車雙保險、CI 分層、
  promote 權限、self-review 留痕、stale_suspect、macro/micro 並陳、
  low_n、保守模式最少集、s04 白名單入 profile、兩套 DoD。
- **兩點補強**:
  1. 突變測試不是遷移時的一次性檢查,應自動化為常駐測試
     (profile 活性金絲雀),防止規則日後被寫回程式。
  2. min 閘門的前提是進閘門的訊號本身乾淨。規格須明文區分
     「關鍵訊號」(進 min,如三種切分依據同界)與「參考訊號」
     (只進 debug breakdown,如指紋距離),否則一個雜訊大的訊號
     會讓系統過度保守到不可用。

## 十三、第三輪(定案輪)

評審第三輪確認無第四輪架構異議,並交付三份形狀的第一版邊界。要點:

1. **兩點補強落成可測契約**:

     加選配靜態檢查——stages 程式碼對業務字面量 deny-list

     「profile 真的在驅動行為」,不是「11108 永遠綠」
   - 訊號分級表:關鍵訊號(進 min 閘門)=三依據同界、錨點品質、
